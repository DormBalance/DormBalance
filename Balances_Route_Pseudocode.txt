================================================================================
  GET HANDLER  -  export async function GET(request: NextRequest)
================================================================================


STEP 1: Read and validate household_id from query params

    const household_id = request.nextUrl.searchParams.get('household_id')

    if !household_id
        return 400: "household_id is required"

    try to convert to BigInt:
        householdID = BigInt(household_id)
    catch:
        return 400: "not a valid household_id"


STEP 2: Wrap everything in a try/catch

    The outer try/catch handles any unexpected error from steps 3-7.
    On catch: console.error the error, return 500.


STEP 3: Load all expenses for the household (with their splits)

    const expenses = await prisma.expenses.findMany({
        where: { household_id: householdID },
        select: {
            payer_user_id: true,
            amount: true,
            splits: {
                select: {
                    user_id: true,
                    amount_to_pay: true,
                    opted_out: true,
                }
            }
        }
    })

    [look up: Prisma select vs include]

    WHY select instead of include?
        We only need a few fields, not the entire expense row.
        "select" picks specific columns; "include" loads full related records.
        Either works, but select is lighter.

    WHY load splits?
        Each expense has a payer (who fronted the money) and splits (who owes
        their share). To compute balances, you need both.


STEP 4: Load all settlements for the household

    const settlements = await prisma.settlements.findMany({
        where: { household_id: householdID },
        select: {
            payer_user_id: true,
            recipient_user_id: true,
            amount: true,
        }
    })

    WHY?
        Settlements reduce debt. If User A owes User B $50, and User A
        settles $30 to User B, the remaining debt is $20.


STEP 5: Build a pairwise balance map

    This is the core math. You need a data structure that tracks:
        "User A owes User B: $X.XX"

    5a. Create a JavaScript Map to hold pairwise balances

        const pairwiseBalances = new Map<string, number>()

        [look up: JavaScript Map, Map.get(), Map.set()]

        The key will be a string like "3->7" meaning user 3 owes user 7.
        The value will be the amount in cents (use cents to avoid
        floating point errors, same as the split helpers do).

    5b. Create a helper function to add to the map

        function addBalance(fromUserId: bigint, toUserId: bigint, cents: number)

        This function takes "fromUser owes toUser X cents" and adds it
        to the pairwise map.

        Pseudocode:
            if fromUserId === toUserId, return (can't owe yourself)

            const key = `${fromUserId}->${toUserId}`
            const current = pairwiseBalances.get(key) ?? 0
            pairwiseBalances.set(key, current + cents)

            ?? is the "nullish coalescing operator" — if get() returns
            undefined, use 0 instead.
            [look up: nullish coalescing operator ??]


STEP 6: Process all expenses and settlements into the map

    6a. Loop over each expense

        for (const expense of expenses)

        For each expense:
            - The PAYER is the person who fronted the money
            - Each SPLIT entry is a person who owes their share

            Skip the expense if payer_user_id is null (shouldn't happen,
            but be safe)

            For each split in expense.splits:
                - Skip if opted_out is true (they don't owe anything)
                - Skip if split.user_id === expense.payer_user_id
                    (the payer's own share — they don't owe themselves)
                - Convert split.amount_to_pay to cents:
                    const cents = Math.round(Number(split.amount_to_pay) * 100)
                - Call addBalance(split.user_id, expense.payer_user_id, cents)
                    meaning: "this split user owes the payer this many cents"

        WHY this logic?
            When User A pays a $100 expense split equally among A, B, C:
            - B owes A $33.33
            - C owes A $33.34
            - A's own split is skipped (A doesn't owe A)

    6b. Loop over each settlement

        for (const settlement of settlements)

        For each settlement:
            - The PAYER is paying back the RECIPIENT
            - This REDUCES what payer owes recipient

            Convert settlement.amount to cents:
                const cents = Math.round(Number(settlement.amount) * 100)
            Call addBalance(settlement.recipient_user_id, settlement.payer_user_id, cents)

            Wait — why recipient -> payer?
                A settlement means "payer paid recipient back".
                If payer owed recipient $50 and paid $30, we need to
                REDUCE what payer owes recipient by $30.
                That's the same as ADDING $30 in the REVERSE direction
                (recipient owes payer $30), which cancels out when we
                net the balances in step 7.

            [look up: double-entry bookkeeping concept]


STEP 7: Net the pairwise balances and build the response

    Now the map has raw entries like:
        "3->7": 5000    (user 3 owes user 7 $50.00 in cents)
        "7->3": 2000    (user 7 owes user 3 $20.00 in cents)

    You need to NET these: 3 actually owes 7 only $30.00.

    7a. Create a Set to track already-processed pairs

        const processedPairs = new Set<string>()

    7b. Create an array for the final pairwise results

        const pairwise: { from: string, to: string, amount: string }[] = []

    7c. Loop over every entry in the map

        for (const [key, amount] of pairwiseBalances)
        [look up: Map iteration, destructuring Map entries]

            Parse the key to get the two user IDs:
                const [fromStr, toStr] = key.split('->')
                [look up: String.split()]

            Build a "canonical pair key" to avoid processing both directions:
                Sort the two IDs so the smaller one is first:
                const pairKey = [fromStr, toStr].sort().join('-')
                [look up: Array.sort(), Array.join()]

                If processedPairs already has pairKey: continue
                Otherwise add pairKey to processedPairs

            Look up both directions:
                const forwardCents = pairwiseBalances.get(`${fromStr}->${toStr}`) ?? 0
                const reverseCents = pairwiseBalances.get(`${toStr}->${fromStr}`) ?? 0

            Compute the net:
                const net = forwardCents - reverseCents

            If net > 0:
                fromStr owes toStr (net / 100) dollars
                push { from: fromStr, to: toStr, amount: (net / 100).toFixed(2) }
            If net < 0:
                toStr owes fromStr (Math.abs(net) / 100) dollars
                push { from: toStr, to: fromStr, amount: (Math.abs(net) / 100).toFixed(2) }
            If net === 0:
                they're even — don't add to the array


STEP 8: Return the response

    return NextResponse.json(
        { success: true, balances: JSONifyBigInt(pairwise) },
        { status: 200 }
    )


STEP 9: Catch block (outer try/catch)

    catch (err) {
        console.error(`Error getting balances for household ${household_id}: `, err)
        return NextResponse.json(
            { error: 'We encountered an error while getting balances' },
            { status: 500 }
        )
    }


================================================================================
  THE MATH EXPLAINED WITH AN EXAMPLE
================================================================================

  Household has 3 users: Alice (1), Bob (2), Charlie (3)

  EXPENSE 1: Alice pays $90 dinner, split equally (each owes $30)
      Split rows: Alice $30, Bob $30, Charlie $30
      Result:
          Bob owes Alice $30      →  addBalance(2, 1, 3000)
          Charlie owes Alice $30  →  addBalance(3, 1, 3000)
          (Alice's own $30 split is skipped)

  EXPENSE 2: Bob pays $60 groceries, split equally (each owes $20)
      Split rows: Alice $20, Bob $20, Charlie $20
      Result:
          Alice owes Bob $20      →  addBalance(1, 2, 2000)
          Charlie owes Bob $20    →  addBalance(3, 2, 2000)
          (Bob's own $20 split is skipped)

  SETTLEMENT: Bob pays Alice $10
      Result:
          Alice owes Bob $10      →  addBalance(1, 2, 1000)
          (reverse direction adds to Alice->Bob, canceling Bob's debt)

  Raw map after processing:
      "2->1": 3000     Bob owes Alice 3000 cents
      "3->1": 3000     Charlie owes Alice 3000 cents
      "1->2": 2000     Alice owes Bob 2000 cents
      "3->2": 2000     Charlie owes Bob 2000 cents
      "1->2": +1000    (settlement adds 1000 more) = 3000 total

  Netting:
      Bob<->Alice:  Bob owes Alice 3000, Alice owes Bob 3000  → net 0, even!
      Charlie<->Alice: Charlie owes Alice 3000, Alice owes Charlie 0 → net 3000
      Charlie<->Bob: Charlie owes Bob 2000, Bob owes Charlie 0 → net 2000

  Final response:
      [
          { "from": "3", "to": "1", "amount": "30.00" },
          { "from": "3", "to": "2", "amount": "20.00" }
      ]

  Charlie owes Alice $30 and Bob $20. Alice and Bob are even.
